# clausr.ai — AI 合规评估智能体

**clausr.ai** 是为认证、检测、检验机构打造的 AI 合规评估平台。它不是一个通用 AI 聊天机器人——它是一个将专业经验结构化沉淀为可复用数字资产、让知识在组织内自由流动、让每一次评估结论天然可审计的专业系统。

---

## 一、核心价值

### 1. 资深审核员专业经验数字化沉淀

认证机构最宝贵的资产不是设备，是人的经验——将这些经验数字化，就可以被规模化复用。clausr.ai 帮助审核员将隐性知识结构化地存储在 **Skills** 中——每一个合规领域的审核方法论，从法规解读到检查要点、从判定逻辑到报告模板，全部显式编码为可维护、可传承、可积累的数字资产。经验被数字化后，成为组织可以积累、可以分享的永久资产。

### 2. 天然审计就绪

每一次评估的每一条结论，都精确追溯到具体法规条款和条款原文。不是"依据相关法规"的模糊表述，而是"依据 UN R48 §5.11，LED 光源必须配备自动调平系统，受检车辆未安装该系统"。引用编译在代码层完成，不经 LLM 生成——杜绝伪造引用、杜绝幻觉条款。监管核查时，审计线索不需要事后整理，它一开始就内建在报告中。

### 3. 一次修正，永久生效

通用 AI 的"学习"是黑盒微调——你不知道它记住了什么，也无法验证它是否真的改了。clausr.ai 的进化是**显式知识修正**：审核员发现某个判定需要调整，直接修改对应 Skill 的规则章节。该 Skill 下所有后续评估自动应用新规则。一处理解纠正，永久全局生效。知识积累可追踪、可对比、可回滚。

### 4. 知识流向需要它的地方

当资深审核员的方法论被完整编码为 Skill 后，知识的价值被放大——不再局限于服务一个人、一个岗位、一个时刻。初级人员输入检测数据，系统按资深审核员的逻辑完成评估，引用正确的法规、执行正确的检查、得出正确的结论。这不是"AI 替代人"，而是"审核员的判断力通过 AI 被分享到需要它的任何环节"。

知识的流向不止于此。检测端发现的问题模式，可以回流到设计端成为改进依据。一家机构积累的经验修正，可以为全行业提供参考。在 clausr.ai 的架构里，知识从有经验的地方流向需要经验的地方——跨人、跨部门、跨机构。

## 二、技术架构：clausr.ai 与通用 Chatbot 的架构级区别

通用 AI 聊天机器人本质上是一个请求-响应黑盒：用户提问 → LLM 生成回答。它不知道步骤在哪里，不知道引用是否真实，不知道谁会纠正它。clausr.ai 从架构上解决了这些问题。

### 2.1 四层分离架构

clausr.ai 将合规评估拆分为四个独立层次，每一层有独立的职责、独立的演进路径：

| 层次 | 职责 | 特征 |
|------|------|------|
| **Capture（采集）** | 多模态证据输入 → 结构化纯文本 | 面向易用性：用户上传 PDF、图片、聊天记录，系统归一化为结构化文本 |
| **Knowledge（知识）** | Skills、法规语料、记忆、进化 | 面向复利：唯一随时间增值的层；技能越多越强，用户越离不开 |
| **Reasoning（推理）** | LLM 管线、工具调用、溯源引用 | 面向信任：决策过程透明，每一步可追溯 |
| **Deliver（交付）** | 模板解析、.docx 输入输出 | 面向可靠：刻意简单，做格式转换，不做智能处理 |

通用 Chatbot 将四个层次压入一次 LLM 调用。clausr.ai 将它们解耦——Knowledge 复利增长不依赖 Reasoning 的模型更新，Deliver 的格式转换不出错因为它不涉及推理。每一层可以独立优化、独立替换。

### 2.2 动态步骤解析 + 类型驱动调度

这是 clausr.ai 与通用 Chatbot 最根本的架构差异。

通用 Chatbot 的处理逻辑是：将一整段系统提示词交给 LLM，期待 LLM 自己"按顺序执行"。这有两个致命缺陷：(1) LLM 可能跳步、合并步骤、擅自改变执行顺序；(2) 无法区分"需要 LLM 推理"和"应该确定性执行"的操作。

clausr.ai 的处理逻辑是：

- **步骤从 Skill 文件动态解析**：每个 Skill 的 `§2 Execution Flow` 定义了该合规领域的检查步骤。这些步骤由 Skill 作者（资深审核员）自由定义——6 步、10 步、不限数量、不限内容。系统启动时动态解析，不是硬编码。
- **类型驱动的通用执行器**：每个步骤被自动分类为三种类型之一，每种类型由专门的执行器处理：

| 步骤类型 | 执行器 | 适用场景 |
|----------|--------|----------|
| `llm` | LLM 推理执行器 | 需要法规理解和判断的步骤，如"识别车辆照明系统配置" |
| `script` | 确定性脚本执行器 | 需要确定性计算的步骤，如加载法规引用文档、解析模板 |
| `llm+tool` | LLM + 工具调用执行器 | 需要 LLM 推理并调用外部工具的场景，如检查合规数值时需要调用 Python 脚本计算 |

- **上下文管线**：步骤之间通过统一的上下文对象传递数据。每个步骤的产出可以写入上下文，后续步骤按需读取。任意步骤可以产出任意结构的数据——不强制固定 schema，由 Skill 作者自由定义步骤间的数据契约。

这意味着：clausr.ai **强制执行步骤顺序**，而不是"提示" LLM 按顺序执行；它**区分推理任务和确定性任务**，而不是把所有事情都扔给 LLM。

### 2.3 内置处理器 —— 确定性操作零 LLM 调用

clausr.ai 在步骤执行前会进行关键词匹配拦截。如果步骤标题匹配到已知的确定性操作模式，系统**跳过 LLM，直接在代码层执行**：

| 内置处理器 | 匹配模式 | 执行方式 |
|------------|----------|----------|
| 引用文档加载 | 标题含 `load/reference/regulation` 关键词 | 按 Skill §6 规则查找并加载法规原文 |
| 引用编译 | 标题含 `compile/citation/source` 关键词 | 遍历所有检查结果，从引用库匹配对应条款，构建引用索引 |
| 结论判定 | 标题含 `verdict/conclusion` 关键词 | 检查结果中任一 FAIL → 总体 FAIL，否则 PASS |

这三类操作零 LLM 调用、零延迟、零 token 费用、零幻觉风险。引用编译尤其关键——通用 Chatbot 的"引用"是 LLM 生成的，可能是编造的。clausr.ai 的引用是在代码层从法规库中精确匹配而来的，每一条都有原文可查。

### 2.4 数值计算走脚本，非 LLM 猜算

合规评估中大量涉及数值比对——光束截止角是否 ≤ 0.57°？色温是否在 500K-6000K 范围内？安装高度是否在最小/最大值之间？

通用 Chatbot 让 LLM 直接做数学计算和数值比较。LLM 是语言模型，不是计算器——它会算错、会遗漏、会在比较时"差不多就行"。

clausr.ai 的做法：LLM 只负责判断"这个检查点适用哪条法规、需要比较什么参数"，实际的数值计算和比较通过 **function calling** 调用确定性 Python 脚本执行。脚本通过 stdin 接收结构化 JSON 输入，stdout 输出结构化结果。执行环境安全隔离（参数化 execFile，无 shell 拼接）。

LLM 做推理，脚本做计算——各司其职。

### 2.5 显式知识进化

通用 AI 产品谈"学习"通常指模型微调或 RAG 优化——本质上是不透明、不可审计的参数调整。

clausr.ai 的进化机制完全不同：
- 审核员在使用中发现某个判定需要调整 → 直接修改对应 Skill 的 `§7 Knowledge & Evolution` 章节
- 系统通过 fuzzy dedup 算法检查新增规则是否与已有规则重复，避免知识膨胀
- 该 Skill 下所有后续评估自动读取更新后的规则
- 进化历史完整记录在 Skill 文件中——什么时候、谁、改了什么、为什么

这是**显式知识管理**，不是黑盒模型微调。审计员可以看到进化过程，监管机构可以审查规则变更，团队可以对知识版本进行对比和回滚。

### 2.6 双轨路由，渐进升级

clausr.ai 的管道系统支持版本共存：v1 通过传统 LLM 流式响应运行，v2 通过上述的分步管道执行。通过参数 `?pipeline=v2` 切换。这确保了：
- 已有 Skill 不受影响，持续运行在验证过的 v1 管线上
- 新 Skill 或主动选择 v2 的客户享受分步管道的可控性和可观测性
- 两条管线共享同一套 Skill 格式，迁移零成本

## 三、商业模式

clausr.ai 以**平台化模式**运营：

**平台使用收费**：按照评估次数或使用量收费。支持**企业私有空间**——认证机构拥有独立的数据空间、Skill 库和评估记录，数据隔离、权限自主；同时提供**公共空间**——行业通用法规库、社区贡献的基准 Skill，降低新用户启动成本。

**审核员技能提升培训**：帮助认证机构的审核员学会编写结构化 Skill。将"我干了三十年审核"转化为可被系统执行的数字化方法论，既是服务，也是能力建设——客户越会写 Skill，对平台的依赖越深。

**数字化转型咨询**：帮助认证检测机构完成从纸质流程到数字化工作流的过渡。包括多模态证据采集方案、报告模板自动化、与现有 LIMS/ERP 系统对接。咨询不依赖 AI，但它是平台落地的必要前提——机构先了解数字化对业务的意义，AI并非只是一个工具，才能让 AI 发挥作用。

## 四、商业价值

### 1. 入驻"工业质算云公共服务平台"

中国联通的工业质算云定位为面向制造业的质检能力公共服务平台。clausr.ai 作为**"检验检测合规智能体"**嵌入该平台，为平台上的制造企业和检测机构提供 AI 驱动的合规评估能力——从产品合规检测到出口认证文件审查，从工厂审核到供应链合规追溯。

### 2. 定义行业标准的机会

当前，AI 在检测/检验/认证（TIC）领域的应用尚无标准——AI Agent 应该以什么方式工作？Skill 应该以什么格式编写？合规评估流程应该如何设计？
检验检测是千行百业的质量基石，AI Agent缺乏质量标准，会在检验检测实践中引入难以预估的风险，进而传导至各行各业

clausr.ai 的 Skill 规范——从步骤定义、法规引用机制到进化规则——有潜力成为 TIC 领域 AI Agent 的事实标准。中国联通作为工业互联网基础设施的构建者，推动这一标准的制定和推广，将产生远超软件采购本身的产业影响力。

### 3. Skills 数据资产是最深的护城河

底层技术不是壁垒。LLM 的能力边界是透明的，多模态输入是成熟的，Pipeline 架构是可复制的。真正的壁垒在于**垂直领域的 Skills 积累**。

每一个细分认证领域——汽车照明、医疗器械、建材防火、食品接触材料——都需要一套专门的 Skill。这套 Skill 里包含了该领域的法规体系理解、检查要点设计、边缘案例经验、判定逻辑和进化历史。这不是靠更好的模型能追上的——任何后来者都需要从头积累这个知识库。

clausr.ai + 中国联通工业质算云的组合，意味着：**平台积累了最大规模的 TIC 领域 Skills 资产，定义了 AI 质检合规 Agent 的工作标准，后来者追赶的不是技术差距，而是知识积累的时间差距。**

---

clausr.ai 不是一个更好的 ChatGPT。它是一个为认证、检测、检验行业从零设计的 AI 合规评估系统——知识结构化、溯源内建、进化显式、标准可定义。
